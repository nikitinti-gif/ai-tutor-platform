# Рабочая программа обучения автоматизации бизнеса

## Назначение

Программа формируется по реальной практике наших проектов, прежде всего VISCONTI AI ERP и AI Tutor Platform. Она превращает выполненную проектную работу в последовательный учебный курс: бизнес-задача → решение → реализация → проверка → повторяемый метод.

Это живой учебный документ, а не теоретический roadmap. Новый урок добавляется только тогда, когда соответствующая работа действительно выполнена или разобрана на реальном проекте.

## Цель обучения

Научиться самостоятельно находить повторяющиеся бизнес-процессы, описывать экспертные правила, проектировать безопасную автоматизацию, собирать рабочие решения на Python и AI и доводить их до проверяемого результата.

## Как пополняется программа

После каждой содержательной проектной сессии проверяются два вопроса:

1. Появился ли новый переносимый навык или метод?
2. Есть ли реальный пример, результат и способ проверки?

Если оба ответа положительные, в программу добавляется или обновляется урок. Незавершённые идеи не отмечаются как освоенные.

## Освоенные уроки

### Урок 1. Источник правды и защита проекта от расползания

**Практический кейс:** AI Tutor Platform.

**Проблема:** длинная разработка и несколько обсуждений приводят к дублирующим модулям, устаревшим планам и потере текущего приоритета.

**Метод:**

1. Хранить один файл состояния проекта.
2. Перед изменением кода читать этот файл и проверять реальную структуру проекта.
3. Сначала искать существующий модуль с нужной ответственностью.
4. Работать над одной функцией за раз.
5. Считать функцию готовой только после проверки пользовательского сценария.

**Реальный пример:** `docs/CLAUDE.md` определяет текущую фазу, известный баг, запрещённые для изменения области и обязательный end-to-end сценарий.

**Практическое задание:** перед следующей задачей сформулировать текущую цель одним предложением, назвать проверяемый результат и перечислить файлы, которые нельзя менять.

**Критерий освоения:** новая задача не создаёт параллельный модуль и не расширяет текущую фазу без решения пользователя.

### Урок 2. Три ловушки проектной автоматизации

**Практический кейс:** AI Tutor Platform и VISCONTI AI ERP.

**Ловушка 1 — расширение раньше ядра:** новые роли, аналитика и AI-функции создают ощущение большого продукта, но не заменяют один полностью работающий пользовательский цикл.

**Рабочее правило:** сначала выбрать центральный сценарий и довести его до end-to-end результата на реальных данных. Следующий модуль начинается только после фиксации проверки.

**Ловушка 2 — потеря контекста:** решения остаются в разных чатах, а следующая сессия начинается с восстановления структуры и повторного обсуждения.

**Рабочее правило:** один источник правды описывает текущую фазу и ограничения; рабочий журнал фиксирует факты каждой завершённой сессии; код подтверждает фактическое состояние.

**Ловушка 3 — преждевременный «финальный код»:** решение, построенное без реальных шаблонов, файлов или интеграций, может быть логически правильным, но неработоспособным в реальной среде.

**Рабочее правило:** разделять статусы:

- `черновик` — предложена структура;
- `прототип` — код написан, но интеграция не проверена;
- `требует проверки` — локальные проверки пройдены, ручной сценарий не завершён;
- `готово` — реальный пользовательский сценарий прошёл end-to-end без блокирующих ошибок.

**Практическое задание:** для текущего сценария AI Tutor Platform записать границы ядра, реальные входные данные, ожидаемый результат и доказательство готовности.

**Критерий освоения:** участник может честно определить статус функции и не расширяет проект до завершения проверяемого ядра.

### Урок 3. Граница данных при подключении внешнего AI

**Практический кейс:** подключение Gemini к AI Tutor Platform.

**Бизнес-проблема:** внешний AI API получает содержимое запроса. Даже если интеграция технически проста, нельзя автоматически считать допустимой передачу ответов учеников, персональных данных или рабочих документов третьей стороне.

**Принятое решение:** сначала тестировать модель только на искусственных обезличенных примерах. В коде должна существовать техническая граница, а не только устная договорённость.

**Реализация:** обычный Telegram-сценарий не отправляет данные в Gemini. Вызов внешней модели разрешён только тестовому запуску с явным признаком `synthetic_test=True`.

**Повторяемый метод:**

1. Назвать внешний сервис и точный состав передаваемых данных.
2. Проверить условия бесплатного и платного тарифов.
3. Получить отдельное решение владельца проекта о допустимой категории данных.
4. Закрепить решение техническим ограничением.
5. Начать с синтетических примеров и только после оценки качества расширять доступ.

**Критерий освоения:** разработчик может показать, где именно в коде блокируется непредусмотренная передача данных, и доказать, что обычный пользовательский сценарий не обходит ограничение.

### Урок 4. Диагностика облачного развёртывания по слоям

**Практический кейс:** перенос AI Tutor Platform на Render Frankfurt.

**Бизнес-проблема:** локальный регион не позволял вызвать Gemini, а прокси и API-агрегатор не дали стабильного доступа. Требовалось вынести выполнение в доступный регион, не превращая перенос в новую архитектурную ветку.

**Принятое решение:** сохранить один Python-бот, добавить серверный webhook-режим и развернуть тестовый Web Service из того же GitHub-репозитория.

**Реализация:**

1. Для локальной разработки сохранён polling, для сервера добавлен Telegram webhook.
2. Добавлен health endpoint, подтверждающий запуск процесса независимо от Telegram.
3. Версия Python зафиксирована в репозитории через `.python-version`.
4. Регистрация webhook вынесена в фоновую задачу с повторными попытками, чтобы временная DNS-ошибка не останавливала сервер.
5. Секреты передаются только через переменные окружения Render.

**Ошибки и диагностика:**

- Ошибка сборки `pydantic-core` оказалась несовместимостью зависимостей с Python 3.14, а не проблемой приложения.
- Ошибка `Failed to resolve host` возникла после сборки, на слое DNS и регистрации Telegram webhook.
- Health-check HTTP 200 доказал, что web service работает даже до успешной регистрации webhook.

**Проверка результата:** Render получил статус Live, порт 10000 открылся, health endpoint ответил 200, Telegram webhook зарегистрировался.

**Повторяемый метод:**

1. Разделять сбой на слои: сборка → запуск процесса → открытие порта → DNS → внешняя интеграция.
2. Читать первую фактическую ошибку в логах, а не повторять deploy без изменения причины.
3. Фиксировать рабочую версию runtime в репозитории.
4. Не позволять временной внешней ошибке блокировать запуск основного процесса.
5. Подтверждать каждый слой отдельным наблюдаемым сигналом.

**Практическое задание:** для следующего deploy заранее выписать по одному сигналу успеха для сборки, процесса, HTTP-сервиса и внешней интеграции.

**Критерий освоения:** участник может определить слой отказа по логу, внести минимальное исправление и доказать восстановление конкретным сигналом.

### Урок 5. Контрольный набор вместо доверия уверенности AI

**Практический кейс:** проверка Gemini в AI Tutor Platform на 15 синтетических решениях.

**Бизнес-проблема:** ответ модели может выглядеть убедительно и содержать высокую уверенность, но оставаться фактически неверным. В первом прогоне Gemini ошибочно отклонил правильный перевод двоичного числа и указал уверенность 1.00.

**Принятое решение:** оценивать AI не по впечатлению от отдельных ответов и не по его собственной уверенности, а по заранее размеченному контрольному набору.

**Проверка результата:** первый прогон обработал все 15 запросов без ошибок API и выявил одно ложное срабатывание. После общего исправления промпта неизменённый набор дал 15/15 без ошибок и расхождений.

**Повторяемый метод:**

1. До запуска зафиксировать входные примеры и ожидаемые результаты.
2. Отдельно считать ошибки транспорта и ошибки качества модели.
3. Сохранять расхождения вместе с объяснением модели.
4. Не заменять правильный эталон ответом AI только из-за высокой уверенности.
5. После исправления повторять тот же контрольный пример и полный набор.

**Практическое задание:** выполнено — объяснение Gemini получено, ошибка классифицирована как неверное рассуждение модели, внесено минимальное исправление промпта и выполнен повторный прогон.

**Критерий освоения:** участник может посчитать качество на размеченном наборе, отличить API-сбой от ложного решения модели и доказать улучшение повторным тестом.

### Урок 6. Поэтапное подключение мультимодального AI

**Практический кейс:** первый vision-сценарий AI Tutor Platform.

**Бизнес-проблема:** подключение фотографий сразу к пользовательскому боту смешивает качество распознавания, ошибки API и риски передачи персональных данных. При сбое становится непонятно, какой слой не работает.

**Принятое решение:** сначала использовать один встроенный синтетический JPEG без персональных данных и отдельную администраторскую команду.

**Реализация:** изображение передаётся inline через `types.Part.from_bytes`, Gemini возвращает ответ по JSON Schema, а обычный путь ученика технически не может вызвать внешний vision-метод.

**Ошибка и исправление:** экспериментальный Interactions API вернул HTTP 400 из-за структуры мультимодального `input`. Только vision-вызов был переведён на `models.generate_content`; проверенный текстовый контур не изменялся.

**Проверка результата:** первый единый vision-запрос дал 4/5 и уверенно додумал решение на размытом изображении. После разделения транскрипции и проверки неизменяемый набор дал 5/5 без API-ошибок; нечитаемый fixture получил `unclear`.

**Повторяемый метод:**

1. Отделить синтетический тестовый канал от реального пользовательского ввода.
2. Проверить доставку файла и формат API до оценки качества модели.
3. При исправлении менять только отказавший интеграционный слой.
4. Подтвердить один end-to-end пример.
5. Затем расширить тестовый набор на правильные, ошибочные и нечитаемые входы.
6. Не передавать условие задачи на этап OCR: иначе модель может заполнить неразличимый текст по смыслу.
7. Для `partial` и `unreadable` останавливать процесс до проверки решения.

**Критерий освоения:** выполнен — участник может показать безопасную границу данных, диагностировать мультимодальный сбой, отделить транскрипцию от оценки, подтвердить benchmark 5/5 и провести загруженное синтетическое фото через весь Telegram-сценарий.

### Урок 7. Временная обработка файла без прикладного хранения

**Практический кейс:** администраторская загрузка синтетического фото в AI Tutor Platform.

**Бизнес-проблема:** для автоматической проверки нужно получить файл из интерфейса, но раннее сохранение изображений создаёт лишний риск, хранилище и обязательства по удалению данных.

**Принятое решение:** Telegram-фото скачивается в `BytesIO`, обрабатывается в памяти фоновой задачи и не записывается приложением в папку, JSON или профиль ученика.

**Контроль доступа:** vision-вызов разрешён только `ADMIN_TELEGRAM_ID` и только для фото с явной подписью `/check_synthetic_photo` и текстом задания.

**Проверка результата:** загруженный синтетический пример был точно транскрибирован, математическая ошибка обнаружена, педагогическая подсказка сформирована; серверный сценарий завершился без API-ошибок.

**Повторяемый метод:**

1. До загрузки определить, нужно ли вообще хранить исходный файл.
2. Если долговременное хранение не требуется, использовать поток или буфер памяти.
3. Ограничить размер файла и круг пользователей до обработки.
4. Не логировать байты, file ID и лишние данные.
5. Разделить временную обработку и последующее решение о сохранении результата.

**Критерий освоения:** участник может реализовать проверяемый файловый сценарий, не создавая неоговорённое прикладное хранилище.

**Правило изменения политики:** политика хранения версионируется и может меняться вместе с проектом. Новая версия принимается до изменения обработки данных, а для уже накопленных записей отдельно выбирается сохранение, миграция или удаление. Расширять состав данных молча запрещено.

### Урок 8. Whitelist вместо сохранения полного AI-ответа

**Практический кейс:** запись синтетического результата AI Tutor Platform в `learning_dna`.

**Бизнес-проблема:** полный результат мультимодальной проверки содержит транскрипцию, подсказки и технические поля. Простое сохранение всего словаря незаметно нарушило бы принятую политику минимизации данных.

**Принятое решение:** хранилище создаёт новую запись только из четырёх явно разрешённых полей, а не удаляет несколько запрещённых полей из исходного объекта.

**Проверка результата:** сервер последовательно сохранил два разных результата без redeploy и после второй записи прочитал журнал размером 2. Автоматический тест передал дополнительные фото, транскрипцию, условие и Telegram ID и подтвердил их отсутствие в JSON.

**Повторяемый метод:**

1. Формализовать список разрешённых полей.
2. Конструировать сохраняемый объект заново по whitelist.
3. Проверять типы и допустимые значения на границе хранилища.
4. Тестировать с намеренно переданными запрещёнными полями.
5. После записи читать данные обратно и показывать наблюдаемый результат.

**Критерий освоения:** выполнен — участник доказал, какие данные сохраняются, какие технически не могут попасть в запись, и что журнал добавляет результаты вместо перезаписи.

### Урок 9. Постепенная замена временного хранилища

**Практический кейс:** перенос только синтетического журнала AI Tutor Platform из временного JSON в PostgreSQL.

**Бизнес-проблема:** полный перенос базы увеличивает риск до завершения ядра, но временный диск Render не позволяет проверить сохранность результатов после redeploy.

**Принятое решение:** переключать один проверенный контракт через `DATABASE_URL`, сохранив JSON fallback и не затрагивая остальные данные проекта.

**Проверка результата:** после подключения PostgreSQL журнал достиг 2 записей; после redeploy Web Service следующая проверка вернула размер 3. Данные пережили замену контейнера приложения.

**Критерий освоения:** выполнен — участник выделил минимальную границу миграции, сохранил обратную совместимость и доказал долговечность данных end-to-end.

### Урок 10. Проверяемое условие раньше AI-проверки

**Практический кейс:** закрытый student-first цикл AI Tutor Platform по теме «Условные операторы».

**Бизнес-проблема:** интерфейс показывал пять заданий, но они были текстами-заглушками без входных данных и ожидаемого поведения. AI мог дать убедительный ответ, однако объективного эталона проверки не существовало.

**Принятое решение:** до вызова Gemini создать одно конкретное задание с однозначным контрактом: программа читает `x` и выводит `positive`, `negative` или `zero`. Реальный вызов оставлен только администратору, а результат сохраняется по whitelist политики v1.

**Проверка результата:** правильная программа получила `correct`; версия без случая `x == 0` получила `has_error` и точную подсказку. Оба назначения прошли `new → submitted → checked`, PostgreSQL-журнал вырос с 3 до 5 записей.

**Повторяемый метод:**

1. Проверить, содержит ли задание конкретные входы, правила и ожидаемый результат.
2. Не запускать AI-оценивание для текста-заглушки.
3. Сначала прогнать заведомо правильный ответ.
4. Затем изменить одну существенную ветку и проверить обнаружение ошибки.
5. Отдельно подтвердить пользовательский статус и долговременную запись.

**Критерий освоения:** выполнен — участник отличил работающий интерфейс от проверяемого учебного контракта и доказал полный цикл на положительном и отрицательном примерах.

### Урок 11. Обратимый human-in-the-loop пилот

**Практический кейс:** реальное фото решения обрабатывается Qwen в памяти, но окончательный ответ остаётся за преподавателем.

**Бизнес-проблема:** нужно проверить, действительно ли AI снимает нагрузку, не открывая ученикам автоматические решения и не строя заранее большой интерфейс согласования.

**Принятое решение:** включать контур отдельным флагом, не сохранять фото на диск, отправлять преподавателю исходное фото и AI-черновик, а ученику сообщать только о передаче на проверку.

**Критерий освоения:** пилот можно включить и выключить одной настройкой; ни один AI-вердикт не становится итоговым без участия преподавателя.

## Ближайшее пополнение

Следующий урок создаётся после практического прогона policy v2 и решения, нужен ли отдельный интерфейс подтверждения преподавателя.

### Урок 12. Структурный JSON ещё не гарантирует смысловую корректность

**Практический кейс:** Gemini правильно поставил `has_error` двоичному
решению, но противоречиво похвалил ошибочно выбранные веса разрядов.

**Бизнес-проблема:** проверка схемы подтверждает наличие полей и допустимый
статус, но не доказывает, что объяснение связано с фактическими данными.

**Принятое решение:** требовать дословное доказательство из входа
(`error_evidence`) и локальное объяснение (`error_explanation`). Приложение
проверяет присутствие доказательства во входе и само формирует безопасный
текст; неподтверждённый вердикт переводится в `unclear`.

**Повторяемый метод:**

1. Отделить вердикт модели от доказательства.
2. Потребовать ссылку на конкретный фрагмент исходных данных.
3. Проверить ссылку детерминированным кодом.
4. Формировать пользовательский текст из проверенных полей.
5. Сохранять исходный вывод отдельно для диагностики.
6. При нарушении контракта выбирать безопасный fallback.

**Критерий освоения:** участник умеет отличить валидацию JSON-схемы от
валидации смысла и строит автоматизацию, которая безопасно отказывает при
неподтверждённом выводе AI.

### Урок 13. Одна очередь — разные политики обработки

**Практический кейс:** преподавательская очередь AI Tutor Platform объединяет
синтетические и реальные работы, но бесплатный Gemini обрабатывает только
синтетические.

**Бизнес-проблема:** разделение на две независимые системы усложнило бы
интерфейс и миграцию, а единый безусловный worker нарушил бы политику данных.

**Принятое решение:** хранить работы в одной таблице, маркировать происхождение
полем `is_synthetic` и применять политику на уровне SQL-запроса worker. Экран
преподавателя показывает оба типа, но сам не запускает AI.

**Повторяемый метод:**

1. Использовать единый жизненный цикл сущности.
2. Добавить явный классификационный признак данных.
3. Фильтровать автоматизацию до извлечения записи из БД.
4. Сделать границу видимой оператору в интерфейсе.
5. Проверить, что read-only экран не создаёт побочных действий.

**Критерий освоения:** участник умеет строить одну операционную очередь с
разными проверяемыми политиками обработки, не дублируя систему целиком.

### Урок 14. Человек завершает автоматизированный процесс

**Практический кейс:** карточка работы преподавателя завершает обработку после
автоматического анализа или ручного просмотра real-фото.

**Бизнес-проблема:** наличие AI-результата в базе ещё не означает завершённый
бизнес-процесс. Нужны исходный документ, наблюдаемый результат и явное действие
ответственного человека.

**Принятое решение:** открыть карточку по устойчивому ID, показать оригинал и
сохранённый анализ, а затем атомарно записать `completed` и время подтверждения.
AI при просмотре повторно не вызывается.

**Повторяемый метод:**

1. Сохранить устойчивый идентификатор процесса.
2. Показать оператору исходные данные и результат автоматизации вместе.
3. Ограничить допустимые переходы статусов на уровне БД.
4. Сделать подтверждение идемпотентным и наблюдаемым.
5. Не запускать дорогую автоматизацию при повторном просмотре.

**Критерий освоения:** участник может провести одну сущность от поступления до
явно подтверждённого терминального статуса без скрытых побочных действий.

### Урок 15. Авторизация не может зависеть от временного диска

**Практический кейс:** после deploy AI Tutor Platform Telegram сохранял старую
клавиатуру teacher, но Render терял JSON-регистрацию, и backend запрещал доступ.

**Бизнес-проблема:** интерфейс создавал видимость полномочий, тогда как источник
авторизации был непостоянным. Временный admin bypass восстанавливал работу, но
не устранял причину.

**Принятое решение:** перенести пользователей и роли в PostgreSQL, оставить
JSON только для локальной разработки, запретить изменение роли через повторную
регистрацию и ограничить создание teacher административным ID.

**Повторяемый метод:**

1. Найти фактический источник решения о доступе.
2. Проверить его поведение после перезапуска и redeploy.
3. Перенести минимальный авторизационный контракт в долговечное хранилище.
4. Сделать повышение роли отдельной привилегированной операцией.
5. Повторить тест доступа до и после redeploy.

**Критерий освоения:** участник отличает сохранённый интерфейс клиента от
серверной авторизации и доказывает долговечность роли сквозным тестом.

### Урок 16. AI-память изменяется только после подтверждённого события

**Практический кейс:** анализ фотографии становится сигналом ДНК знаний только
после завершения проверки преподавателем.

**Бизнес-проблема:** черновик AI может быть ошибочным, а повторное нажатие —
случайным. Если каждый вызов напрямую меняет профиль, аналитика быстро теряет
доверие.

**Принятое решение:** обновлять профиль и терминальный статус работы в одной
транзакции, блокируя строку процесса и проверяя её предыдущий статус.

**Повторяемый метод:**

1. Отделить результат автоматизации от подтверждённого бизнес-события.
2. Выбрать единственную точку записи производного профиля.
3. Объединить изменение статуса и профиля одной транзакцией.
4. Заблокировать повторный эффект проверкой терминального статуса.
5. Показать оператору наблюдаемое подтверждение изменения.

**Критерий освоения:** участник умеет построить идемпотентное обновление
производной аналитики после человеческого подтверждения.

### Урок 17. Наблюдаемая аналитика без повторного AI-вызова

**Практический кейс:** преподаватель открывает накопленную ДНК ученика как
операционную карточку.

**Бизнес-проблема:** скрытая аналитика не помогает принимать решения, а её
повторная генерация моделью при каждом просмотре повышает стоимость и может
изменять формулировки без новых данных.

**Принятое решение:** строить детерминированное представление из сохранённых и
подтверждённых сигналов: темы успехов и ошибок, навыки, фокус и рекомендация.

**Повторяемый метод:**

1. Читать подтверждённое состояние из источника правды.
2. Агрегировать только фактически сохранённые сигналы.
3. Явно показывать отсутствие данных.
4. Не обращаться к AI при обычном просмотре карточки.
5. Проверять, что оператор видит основание следующего решения.

**Критерий освоения:** участник умеет превратить накопленные события в
стабильную и проверяемую карточку для оператора.
