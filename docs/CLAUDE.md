## Что это за проект
Telegram-бот AI-репетитора по информатике. Три роли: student, teacher, parent.
Стек: Python, aiogram, JSON-хранилище (database.json, Postgres — в будущем).
Параллельно есть слой автоматизации на n8n (5 воркфлоу) — статус: не финализирован,
решение "Python-бот ИЛИ n8n" ещё не принято.

## Текущая фаза разработки
Сейчас: **Фаза 1 — проверка реального AI на синтетических решениях.**
В `src/ai_engine/llm_client.py` добавлен реальный вызов Gemini API через
`google-genai`. После ответа Google `404 model no longer available to new users`
модель заменена с `gemini-2.5-flash-lite` на `gemini-3.1-flash-lite`.

Текущий статус: локальный API-вызов дошёл до Google, но был заблокирован по
региону. Попытки через локальный HTTP/SOCKS5-прокси зависли на
TLS-рукопожатии. Первый deploy на Render Frankfurt не собрал зависимости,
потому что Render выбрал Python 3.14, а старый `pydantic-core` попытался
собираться из Rust-исходников. После фиксации Python 3.12 сборка прошла, но
процесс упал: Telegram ещё не мог разрешить новый hostname Render во время
ранней регистрации webhook. Подготовлена фоновая регистрация с повторными
попытками, которая не блокирует открытие HTTP-порта; повторный deploy ещё не
выполнен. Фаза 1 не завершена.

Политика данных на этом этапе: Gemini разрешён ТОЛЬКО для искусственных
обезличенных примеров. Обычный Telegram-сценарий не отправляет текст во внешний
API; внешний вызов требует явного `synthetic_test=True`. Реальные ответы,
имена, Telegram ID и фотографии учеников передавать запрещено.

## Состояние основного бага после синхронизации

`LLMClient.check_homework_text(text)` уже присутствует в `llm_client.py`, поэтому
ранее описанное несовпадение интерфейса устранено в коде.

Фаза 0 завершена ручным прогоном 2026-07-13: ученик отправил текст решения,
проверка вернула результат, педагогический движок сформировал ответ, бот не упал
на основном сценарии. Трассировка `CancelledError` → `KeyboardInterrupt`
появилась только после ручной остановки polling через `Ctrl+C` и не является
ошибкой проверки домашнего задания.

Следующий шаг — синхронизировать и отправить изменения в GitHub, создать
Render Web Service из `render.yaml`, проверить health endpoint и регистрацию
Telegram webhook. Бесплатный Render используется только как тестовая среда:
его файловая система непостоянна, поэтому реальные данные учеников там хранить
запрещено.

## Педагогический движок (концепция готова, реализация — упрощённая)
`src/pedagogy/engine.py` реализует цепочку:
Observation → Evidence → Reasoning → Decision → Action Plan
(файлы `observations.py`, `evidence.py`, `reasoning.py`, `decisions.py`,
`actions.py`). Сейчас это простые правила, не LLM. Это ок для текущей фазы —
не превращать сразу в AI-вызовы, сначала стабилизировать homework_checker.

## Правила при добавлении кода
1. Один модуль/фича за раз. Перед созданием нового файла — проверить (grep/поиск),
   нет ли уже похожего с той же ответственностью. Если есть — дорабатывать его,
   не создавать параллельный (так появился student_brain рядом с learning_dna).
2. Не писать документацию/ADR наперёд для нереализованного функционала.
   Пустые доки (`docs/Vision.md`, `docs/Roadmap.md`, `docs/MVP.md`,
   `docs/architecture/EVENT_CATALOG.md`, `PEDAGOGICAL_POLICIES.md`, `Trajectory.md`)
   заполняются постфактум, когда фича реально сделана — не раньше.
3. Новый код в `src/ai_engine/` должен реально дёргать LLM API, а не
   имитировать через if/else по ключевым словам.
4. Каждая фича считается готовой только после ручного end-to-end прогона
   (см. ниже), а не когда код просто написан.

## Известные антипаттерны ведения проекта

### 1. Расширение раньше завершения ядра

Раньше новые роли, аналитика и AI-функции появлялись до полностью работающего пользовательского цикла.

Правило: сначала завершить один основной student-first цикл — получить задание → отправить решение → получить реальную AI-проверку → сохранить результат. Новые роли, аналитика и функции не добавляются, пока текущий цикл не прошёл end-to-end проверку.

### 2. Потеря контекста между сессиями

Раньше приходилось заново восстанавливать структуру файлов, принятые решения и текущее состояние проекта.

Правило: в начале рабочей сессии читать этот файл и проверять реальный код; в конце содержательной сессии обновлять `docs/WORK_LOG.md`. История чатов не заменяет источник правды и состояние репозитория.

### 3. Ожидание «финального кода» слишком рано

Без проверки реальных файлов, шаблонов, данных и интеграций решение может выглядеть завершённым, но ломаться при запуске.

Правило: не называть код финальным до проверки входных файлов, контрактов модулей, конфигурации и ручного пользовательского сценария. Если интеграционная проверка не выполнена, статус должен быть «прототип» или «требует проверки».

## Как проверять, что всё работает (ручной прогон)
`python bot.py` → сценарий ученика:
`/start` → выбрать роль student → получить задание → отправить решение текстом
→ получить результат проверки без ошибок в консоли.

## Автоматическое ведение рабочих документов

После каждой содержательной сессии, в которой реально изменён или проверен проект, ИИ-ассистент должен:

1. Добавить краткую фактическую запись в `docs/WORK_LOG.md`.
2. Обновить `docs/BUSINESS_AUTOMATION_LEARNING_PROGRAM.md`, только если появился новый переносимый навык, подтверждённый практикой.
3. Обновить `docs/BACKLOG.md`, Release-документы или `NOT_NOW.md`, только если фактически изменился статус соответствующей работы.
4. Не отмечать функцию завершённой без указанной проверки; при отсутствии ручного end-to-end прогона писать «требует проверки».
5. Не создавать записи после простых обсуждений, если код, данные, решение или проверяемое состояние проекта не изменились.

Это правило действует во время рабочих сессий с доступом к файлам проекта. Оно не означает фоновое обновление документов без запущенной сессии.
