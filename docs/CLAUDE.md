## Что это за проект
Telegram-бот AI-репетитора по информатике. Три роли: student, teacher, parent.
Стек: Python, aiogram, JSON-хранилище (database.json, Postgres — в будущем).
Параллельно есть слой автоматизации на n8n (5 воркфлоу) — статус: не финализирован,
решение "Python-бот ИЛИ n8n" ещё не принято.

## Состояние основного бага после синхронизации
По контрольному чтению файлов от 2026-07-13 метод
`LLMClient.check_homework_text(text)` уже присутствует в `llm_client.py`, поэтому
ранее описанное несовпадение интерфейса устранено в коде.

Фаза 0 завершена ручным прогоном 2026-07-13: ученик отправил текст решения,
проверка вернула результат, педагогический движок сформировал ответ, бот не упал
на основном сценарии. Трассировка `CancelledError` → `KeyboardInterrupt`
появилась только после ручной остановки polling через `Ctrl+C` и не является
ошибкой проверки домашнего задания.

Лог Render подтвердил ошибку рассуждения модели: Gemini заявил о неверных
степенях двойки, хотя сокращённая сумма `16 + 4 + 2` верна. Системный промпт
теперь требует независимо перепроверять вычисления и допускает отсутствие
нулевых слагаемых в позиционных системах счисления; повторный прогон 15/15
подтвердил исправление без регрессий на контрольном наборе.

Следующий шаг — отдельно согласовать границу данных и критерий следующей фазы.
До этого реальные ответы и фотографии учеников в Gemini не отправляются.
Бесплатный Render используется только как тестовая среда: его файловая система
непостоянна, поэтому реальные данные учеников там хранить запрещено.

Для первого шага Фазы 2 подготовлена администраторская команда
`/gemini_image_eval`. Она берёт только фиксированное искусственное изображение
`tests/fixtures/synthetic_loop_correct.jpg`, передаёт его Gemini как inline
JPEG и проверяет структурированный ответ. Обычные фотографии из Telegram этим
кодом не принимаются и во внешний API не отправляются. Статус прототипа:
первый серверный запрос дошёл до Gemini, но экспериментальный Interactions API
вернул HTTP 400 из-за неверной структуры мультимодального `input`. Только
vision-метод переведён на `models.generate_content` с inline image и JSON
Schema; текстовый Interactions-контур 15/15 не изменён. Повторный серверный
прогон успешно завершён: Gemini прочитал синтетическое рукописное решение цикла,
вернул `correct` с уверенностью 1.00 и дал предметную обратную связь.

Фаза 2 остаётся в работе: подтверждён один ясный правильный JPEG, но ещё не
проверены ошибочные и намеренно нечитаемые изображения. Первый пример повторно
дал тот же корректный результат, что подтверждает стабильность базового кейса.
Подготовлен неизменяемый набор из 5 синтетических изображений: два цикла, два
перевода из двоичной системы и одно намеренно нечитаемое решение; ожидаемые
статусы покрывают `correct`, `has_error` и `unclear`. Команда
`/gemini_image_eval` запускает весь набор и формирует сводку. Первый серверный
прогон обработал 5/5 изображений без API-ошибок; 4 статуса совпали с эталоном.
`unreadable_handwritten_solution` ожидал `unclear`, но Gemini вернул `correct`
с уверенностью 0.90. `feedback` заявил о считывании числа и умножении, хотя эти
операции нельзя надёжно прочитать на fixture: модель достроила решение из
известного условия.

Подготовлено защитное разделение на два вызова. Первый vision-вызов не получает
условие задачи и возвращает только буквальную транскрипцию, `legibility` и
уверенность. При `partial` или `unreadable` локальный код принудительно возвращает
`unclear` и не запускает проверку решения. Только полностью `readable`
транскрипция передаётся в проверенный текстовый контур. Повторный серверный
benchmark дал 5/5 без API-ошибок и расхождений, включая ожидаемый `unclear` для
размытого fixture. Синтетический vision-контур проверен.

Фаза 2 целиком ещё не завершена: Telegram не принимает загруженное изображение
для AI-проверки, работает только закрытая команда со встроенными fixture.
Следующий шаг требует отдельного решения о допустимых данных и тестовом канале
загрузки. Реальные фотографии учеников по-прежнему запрещены.

Решение принято: подготовлен администраторский канал загрузки только явно
синтетических фото. Администратор отправляет JPEG как Telegram-фото с подписью
`/check_synthetic_photo текст задания`. Бот скачивает файл в `BytesIO`, не пишет
его в `uploads`, JSON или на диск, запускает защищённую двухэтапную проверку и
возвращает читаемость, транскрипцию и педагогический результат. Ограничение —
5 Мбайт. Обычные фото без команды не попадают в этот AI-обработчик. Канал
прошёл серверный end-to-end прогон: загруженный синтетический пример с ошибкой
`10110₂ = 16 + 8 + 2 = 26₁₀` был распознан как `readable`, точно транскрибирован
и получил ожидаемый `has_error` с корректной подсказкой.

Техническое ядро Фазы 2 завершено в согласованной безопасной границе:
синтетический benchmark 5/5 и администраторская загрузка подтверждены. Это не
разрешает обработку реальных фотографий учеников. Пользовательский rollout,
согласие, сроки хранения и запись результатов требуют отдельного решения.

## Правила при добавлении кода
1. Один модуль/фича за раз. Перед созданием нового файла — проверить (grep/поиск),
   нет ли уже похожего с той же ответственностью. Если есть — дорабатывать его,
   не создавать параллельный (так появился student_brain рядом с learning_dna).
2. Не писать документацию/ADR наперёд для нереализованного функционала.
   Пустые доки (`docs/Vision.md`, `docs/Roadmap.md`, `docs/MVP.md`,
   `docs/architecture/EVENT_CATALOG.md`, `PEDAGOGICAL_POLICIES.md`, `Trajectory.md`)
   заполняются постфактум, когда фича реально сделана — не раньше.
3. Новый код в `src/ai_engine/` должен реально дёргать LLM API, а не
   имитировать через if/else по ключевым словам.
4. Каждая фича считается готовой только после ручного end-to-end прогона
   (см. ниже), а не когда код просто написан.

## Известные антипаттерны ведения проекта

### 1. Расширение раньше завершения ядра

Раньше новые роли, аналитика и AI-функции появлялись до полностью работающего пользовательского цикла.

Правило: сначала завершить один основной student-first цикл — получить задание → отправить решение → получить реальную AI-проверку → сохранить результат. Новые роли, аналитика и функции не добавляются, пока текущий цикл не прошёл end-to-end проверку.

### 2. Потеря контекста между сессиями

Раньше приходилось заново восстанавливать структуру файлов, принятые решения и текущее состояние проекта.

Правило: в начале рабочей сессии читать этот файл и проверять реальный код; в конце содержательной сессии обновлять `docs/WORK_LOG.md`. История чатов не заменяет источник правды и состояние репозитория.

### 3. Ожидание «финального кода» слишком рано

Без проверки реальных файлов, шаблонов, данных и интеграций решение может выглядеть завершённым, но ломаться при запуске.

Правило: не называть код финальным до проверки входных файлов, контрактов модулей, конфигурации и ручного пользовательского сценария. Если интеграционная проверка не выполнена, статус должен быть «прототип» или «требует проверки».

## Как проверять, что всё работает (ручной прогон)
`python bot.py` → сценарий ученика:
`/start` → выбрать роль student → получить задание → отправить решение текстом
→ получить результат проверки без ошибок в консоли.

## Автоматическое ведение рабочих документов

После каждой содержательной сессии, в которой реально изменён или проверен проект, ИИ-ассистент должен:

1. Добавить краткую фактическую запись в `docs/WORK_LOG.md`.
2. Обновить `docs/BUSINESS_AUTOMATION_LEARNING_PROGRAM.md`, только если появился новый переносимый навык, подтверждённый практикой.
3. Обновить `docs/BACKLOG.md`, Release-документы или `NOT_NOW.md`, только если фактически изменился статус соответствующей работы.
4. Не отмечать функцию завершённой без указанной проверки; при отсутствии ручного end-to-end прогона писать «требует проверки».
5. Не создавать записи после простых обсуждений, если код, данные, решение или проверяемое состояние проекта не изменились.

Это правило действует во время рабочих сессий с доступом к файлам проекта. Оно не означает фоновое обновление документов без запущенной сессии.
