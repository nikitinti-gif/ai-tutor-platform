## Что это за проект
Telegram-бот AI-репетитора по информатике. Три роли: student, teacher, parent.
Стек: Python, aiogram, JSON-хранилище (database.json, Postgres — в будущем).
Параллельно есть слой автоматизации на n8n (5 воркфлоу) — статус: не финализирован,
решение "Python-бот ИЛИ n8n" ещё не принято.

## Новая продуктовая граница от 2026-07-18

Основной пользователь загрузки — совершеннолетний родитель или законный
представитель, а получатель AI-анализа — учитель. Ребёнок не получает аккаунт,
чат, подсказки, транскрипцию, AI-оценку или доступ к модели. Родитель
фотографирует выполненную работу и видит только подтверждение приёма.

Целевой цикл: родитель → локальная проверка качества фото → защищённая очередь
→ OCR/Vision/LLM → строгая валидация → Learning DNA → готовый анализ и
следующие задания учителю. Рутинная обработка автоматическая; учитель
вмешивается только в сомнительные случаи. До отдельного разрешения данных и
платного Gemini реальные детские работы во внешний AI не отправляются.

Текущий реализуемый срез ограничен каналом приёма: родительская роль отправляет
фото, приложение локально проверяет разрешение, яркость, контраст и резкость,
плохой снимок просит переснять, а принятый передаёт преподавателю без AI-вызова.
Статус этого среза — прототип, требующий ручного end-to-end прогона.

Очередь приёма от 2026-07-18: качественное фото сначала записывается в
PostgreSQL-таблицу `homework_submissions`, и только затем родитель получает
подтверждение. Хранятся Telegram `file_id`, идентификатор родителя, размеры,
метрики качества, статус обработки и служебные временные поля; сам файл в БД
не копируется. Статусы: `accepted`, `processing`, `teacher_review`,
`completed`, `failed`. После семейной привязки новая запись получает
`student_telegram_id`; старые записи с `NULL` автоматически не назначаются.

Привязка семьи от 2026-07-18: преподаватель создаёт для зарегистрированного
ученика одноразовый восьмисимвольный буквенно-цифровой код сроком на 30 минут.
В PostgreSQL хранится только SHA-256-хеш кода. Родитель вводит код один раз,
после чего создаётся связь `parent_telegram_id → student_telegram_id`.
В пилоте у одного родителя может быть один активный ребёнок; повторная
привязка осознанно заменяет предыдущую. Загрузка работы без связи блокируется.
Новые работы сохраняют реальный `student_telegram_id` для будущего Learning
DNA. Работа, принятая до внедрения связи, остаётся с `NULL` и автоматически
никуда не назначается.

Однопользовательский тестовый режим от 2026-07-18 доступен только
`ADMIN_TELEGRAM_ID`: кнопка `🧪 Тестовая семья` создаёт код для синтетического
ученика с отрицательным ID, а `🧪 Сдать тестовую работу` запускает тот же
родительский канал приёма без смены постоянной роли администратора. Такие
записи получают в PostgreSQL `is_synthetic=TRUE`; существующая таблица
обновляется через `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`. Этот режим не
создаёт фиктивного Telegram-пользователя и не должен использоваться для
реальных детских работ.

Синтетический Gemini-worker от 2026-07-18 включается переменной
`SYNTHETIC_GEMINI_WORKER_ENABLED`. Он атомарно выбирает только записи
`status='accepted' AND is_synthetic=TRUE` через `FOR UPDATE SKIP LOCKED`,
скачивает фото по Telegram `file_id` и запускает уже проверенный
`check_homework_image(..., synthetic_test=True, provider_name='gemini')`.
Реальные работы SQL-фильтр не выбирает. Результат сохраняется как JSONB,
статус меняется на `teacher_review`, а анализ отправляется только
`ADMIN_TELEGRAM_ID`. Максимум две попытки; затем статус `failed`, ошибка
остаётся в БД, преподаватель получает уведомление. Если условие задачи не
видно на фото, worker не придумывает его и допускает результат `unclear`.
Один полный анализ использует два вызова Gemini: распознавание изображения и
проверку транскрипции.

Семантический контракт от 2026-07-19: для `has_error` провайдер обязан вернуть
`error_evidence` — дословный самодостаточный фрагмент транскрипции — и
`error_explanation` с локальной причиной. Приложение проверяет наличие этого
фрагмента в решении. Если доказательство отсутствует или придумано, результат
понижается до `unclear` с нулевой уверенностью. Показанный преподавателю
`feedback` строится заново из доказательства и объяснения; исходный текст
провайдера хранится в `provider_feedback` только для диагностики. Это не
создаёт дополнительного AI-вызова.

Граница бесплатного Gemini подтверждена 2026-07-19 по актуальным условиям
Google: отсутствие детского аккаунта решает возрастной доступ, но бесплатный
контур не используется для реальных работ. В проекте действует разделение:
`is_synthetic=TRUE` автоматически обрабатывается Gemini, а
`is_synthetic=FALSE` сохраняется для преподавателя без внешнего AI. Переход
реальных работ в Gemini потребует Paid Service и отдельного включения политики;
согласие родителя само по себе не меняет условия провайдера.

Преподавательская очередь от 2026-07-19 доступна через `📸 Проверки ИИ`.
Она читает последние 10 записей PostgreSQL, раздельно считает synthetic и real,
показывает статусы `accepted`, `processing`, `teacher_review`, `completed`,
`failed`, краткий AI-результат и техническую ошибку. Кнопка не запускает AI и
доступна только зарегистрированной роли `teacher`.

## Состояние основного бага после синхронизации
По контрольному чтению файлов от 2026-07-13 метод
`LLMClient.check_homework_text(text)` уже присутствует в `llm_client.py`, поэтому
ранее описанное несовпадение интерфейса устранено в коде.

Фаза 0 завершена ручным прогоном 2026-07-13: ученик отправил текст решения,
проверка вернула результат, педагогический движок сформировал ответ, бот не упал
на основном сценарии. Трассировка `CancelledError` → `KeyboardInterrupt`
появилась только после ручной остановки polling через `Ctrl+C` и не является
ошибкой проверки домашнего задания.

Лог Render подтвердил ошибку рассуждения модели: Gemini заявил о неверных
степенях двойки, хотя сокращённая сумма `16 + 4 + 2` верна. Системный промпт
теперь требует независимо перепроверять вычисления и допускает отсутствие
нулевых слагаемых в позиционных системах счисления; повторный прогон 15/15
подтвердил исправление без регрессий на контрольном наборе.

Следующий шаг — отдельно согласовать границу данных и критерий следующей фазы.
До этого реальные ответы и фотографии учеников в Gemini не отправляются.
Бесплатный Render используется только как тестовая среда: его файловая система
непостоянна, поэтому реальные данные учеников там хранить запрещено.

Для первого шага Фазы 2 подготовлена администраторская команда
`/gemini_image_eval`. Она берёт только фиксированное искусственное изображение
`tests/fixtures/synthetic_loop_correct.jpg`, передаёт его Gemini как inline
JPEG и проверяет структурированный ответ. Обычные фотографии из Telegram этим
кодом не принимаются и во внешний API не отправляются. Статус прототипа:
первый серверный запрос дошёл до Gemini, но экспериментальный Interactions API
вернул HTTP 400 из-за неверной структуры мультимодального `input`. Только
vision-метод переведён на `models.generate_content` с inline image и JSON
Schema; текстовый Interactions-контур 15/15 не изменён. Повторный серверный
прогон успешно завершён: Gemini прочитал синтетическое рукописное решение цикла,
вернул `correct` с уверенностью 1.00 и дал предметную обратную связь.

Фаза 2 остаётся в работе: подтверждён один ясный правильный JPEG, но ещё не
проверены ошибочные и намеренно нечитаемые изображения. Первый пример повторно
дал тот же корректный результат, что подтверждает стабильность базового кейса.
Подготовлен неизменяемый набор из 5 синтетических изображений: два цикла, два
перевода из двоичной системы и одно намеренно нечитаемое решение; ожидаемые
статусы покрывают `correct`, `has_error` и `unclear`. Команда
`/gemini_image_eval` запускает весь набор и формирует сводку. Первый серверный
прогон обработал 5/5 изображений без API-ошибок; 4 статуса совпали с эталоном.
`unreadable_handwritten_solution` ожидал `unclear`, но Gemini вернул `correct`
с уверенностью 0.90. `feedback` заявил о считывании числа и умножении, хотя эти
операции нельзя надёжно прочитать на fixture: модель достроила решение из
известного условия.

Подготовлено защитное разделение на два вызова. Первый vision-вызов не получает
условие задачи и возвращает только буквальную транскрипцию, `legibility` и
уверенность. При `partial` или `unreadable` локальный код принудительно возвращает
`unclear` и не запускает проверку решения. Только полностью `readable`
транскрипция передаётся в проверенный текстовый контур. Повторный серверный
benchmark дал 5/5 без API-ошибок и расхождений, включая ожидаемый `unclear` для
размытого fixture. Синтетический vision-контур проверен.

Фаза 2 целиком ещё не завершена: Telegram не принимает загруженное изображение
для AI-проверки, работает только закрытая команда со встроенными fixture.
Следующий шаг требует отдельного решения о допустимых данных и тестовом канале
загрузки. Реальные фотографии учеников по-прежнему запрещены.

Решение принято: подготовлен администраторский канал загрузки только явно
синтетических фото. Администратор отправляет JPEG как Telegram-фото с подписью
`/check_synthetic_photo текст задания`. Бот скачивает файл в `BytesIO`, не пишет
его в `uploads`, JSON или на диск, запускает защищённую двухэтапную проверку и
возвращает читаемость, транскрипцию и педагогический результат. Ограничение —
5 Мбайт. Обычные фото без команды не попадают в этот AI-обработчик. Канал
прошёл серверный end-to-end прогон: загруженный синтетический пример с ошибкой
`10110₂ = 16 + 8 + 2 = 26₁₀` был распознан как `readable`, точно транскрибирован
и получил ожидаемый `has_error` с корректной подсказкой.

Техническое ядро Фазы 2 завершено в согласованной безопасной границе:
синтетический benchmark 5/5 и администраторская загрузка подтверждены. Это не
разрешает обработку реальных фотографий учеников. Пользовательский rollout,
согласие, сроки хранения и запись результатов требуют отдельного решения.

## Текущая политика хранения — версия 1

- Исходные изображения приложение не сохраняет.
- Транскрипция используется только во время проверки и не записывается в базу.
- На первом шаге сохраняется только структурированный результат синтетического
  администраторского теста: тема, статус, уверенность и тип ошибки.
- Реальные результаты учеников не сохраняются без отдельного явного решения.
- Политику можно изменить по ходу проекта, но только до соответствующего
  изменения кода и обработки новых данных. Изменение фиксируется в этом файле и
  `WORK_LOG.md`; для ранее созданных записей отдельно принимается решение:
  сохранить, мигрировать или удалить. Молчаливое расширение состава данных
  запрещено.

Реализация политики v1 подготовлена внутри существующего `learning_dna`, без
второго профиля или нового AI-движка. Зарезервированный ключ
`_synthetic_admin_checks` содержит список записей со строгим whitelist:
`topic`, `status`, `confidence`, `error_type`. Репозиторий конструирует запись
заново, поэтому фото, транскрипция, условие, Telegram ID и любые дополнительные
поля результата отбрасываются. Сохранение вызывается только после проверки
загруженного администратором синтетического фото; fixture benchmark и обычные
пользовательские сценарии ничего не записывают. Ответ бота показывает сохранённые
поля и число записей после чтения журнала обратно.

Ограничение среды: `database.json` на бесплатном Render непостоянен и может быть
сброшен при redeploy или перезапуске. На этом шаге проверяется контракт и поток
сохранения, а не долговременная надёжность. Первый серверный end-to-end прогон
успешен: для синтетической ошибки систем счисления сохранены только
`topic=Системы счисления`, `status=has_error`, `confidence=1.00`,
`error_type=calculation_error`; чтение после записи вернуло размер журнала 1.
Фото, транскрипция и условие в запись не включались. Для проверки накопления
без redeploy выполнена вторая синтетическая проверка: правильный пример получил
`topic=Системы счисления`, `status=correct`, `confidence=1.00`,
`error_type=None`; чтение после записи вернуло размер журнала 2. Контракт
добавления и минимизации синтетических результатов подтверждён end-to-end.

Наблюдение OCR: во второй транскрипции нижние индексы были потеряны и отображены
как `101102` и `2210`, хотя итоговая классификация правильного решения не
изменилась. Это не попало в хранилище по политике v1, но остаётся известным
ограничением качества транскрипции математических обозначений.

Подготовлен следующий backend только для синтетического журнала: при наличии
`DATABASE_URL` четыре разрешённых поля атомарно добавляются в JSONB-массив
PostgreSQL; без переменной окружения сохраняется прежний JSON-режим для локальной
разработки. Таблица создаётся автоматически. Остальные разделы `database.json`
не мигрируются и продолжают работать как раньше. Две временные тестовые записи
Render автоматически не переносятся: после подключения базы контракт нужно
повторно подтвердить двумя синтетическими проверками с redeploy между ними.
Для бесплатной PostgreSQL установлен внутренний контрольный дедлайн —
2026-08-04 (23-й день теста): проверить результаты и принять решение об
обновлении существующей базы до платного тарифа до истечения бесплатного срока.

## Правила при добавлении кода
1. Один модуль/фича за раз. Перед созданием нового файла — проверить (grep/поиск),
   нет ли уже похожего с той же ответственностью. Если есть — дорабатывать его,
   не создавать параллельный (так появился student_brain рядом с learning_dna).
2. Не писать документацию/ADR наперёд для нереализованного функционала.
   Пустые доки (`docs/Vision.md`, `docs/Roadmap.md`, `docs/MVP.md`,
   `docs/architecture/EVENT_CATALOG.md`, `PEDAGOGICAL_POLICIES.md`, `Trajectory.md`)
   заполняются постфактум, когда фича реально сделана — не раньше.
3. Новый код в `src/ai_engine/` должен реально дёргать LLM API, а не
   имитировать через if/else по ключевым словам.
4. Каждая фича считается готовой только после ручного end-to-end прогона
   (см. ниже), а не когда код просто написан.

## Известные антипаттерны ведения проекта

### 1. Расширение раньше завершения ядра

Раньше новые роли, аналитика и AI-функции появлялись до полностью работающего пользовательского цикла.

Правило: сначала завершить один основной student-first цикл — получить задание → отправить решение → получить реальную AI-проверку → сохранить результат. Новые роли, аналитика и функции не добавляются, пока текущий цикл не прошёл end-to-end проверку.

### 2. Потеря контекста между сессиями

Раньше приходилось заново восстанавливать структуру файлов, принятые решения и текущее состояние проекта.

Правило: в начале рабочей сессии читать этот файл и проверять реальный код; в конце содержательной сессии обновлять `docs/WORK_LOG.md`. История чатов не заменяет источник правды и состояние репозитория.

### 3. Ожидание «финального кода» слишком рано

Без проверки реальных файлов, шаблонов, данных и интеграций решение может выглядеть завершённым, но ломаться при запуске.

Правило: не называть код финальным до проверки входных файлов, контрактов модулей, конфигурации и ручного пользовательского сценария. Если интеграционная проверка не выполнена, статус должен быть «прототип» или «требует проверки».

## Как проверять, что всё работает (ручной прогон)
`python bot.py` → сценарий ученика:
`/start` → выбрать роль student → получить задание → отправить решение текстом
→ получить результат проверки без ошибок в консоли.

Закрытый синтетический прогон выполняется администратором:
`/student_cycle_demo` → конкретное задание по условным операторам →
`📸 Проверить решение` → текстовый код → реальная проверка Gemini → статус
`checked` → четыре разрешённых поля в PostgreSQL. Обычный пользователь не может
включить `synthetic_test=True`; реальный rollout по-прежнему закрыт.

Сравнительный текстовый benchmark запускается администратором командой
`/provider_eval`. Gemini, Mistral и YandexGPT получают одинаковые 15
синтетических заданий. Для Mistral нужны `MISTRAL_API_KEY` и необязательный
`MISTRAL_MODEL`; для YandexGPT — `YANDEX_API_KEY`, `YANDEX_FOLDER_ID` и
необязательный `YANDEX_MODEL`. Адаптеры Mistral и YandexGPT используют
OpenAI-совместимый Chat Completions API, но по-прежнему отклоняют любой вызов
без `synthetic_test=True`.

## Автоматическое ведение рабочих документов

После каждой содержательной сессии, в которой реально изменён или проверен проект, ИИ-ассистент должен:

1. Добавить краткую фактическую запись в `docs/WORK_LOG.md`.
2. Обновить `docs/BUSINESS_AUTOMATION_LEARNING_PROGRAM.md`, только если появился новый переносимый навык, подтверждённый практикой.
3. Обновить `docs/BACKLOG.md`, Release-документы или `NOT_NOW.md`, только если фактически изменился статус соответствующей работы.
4. Не отмечать функцию завершённой без указанной проверки; при отсутствии ручного end-to-end прогона писать «требует проверки».
5. Не создавать записи после простых обсуждений, если код, данные, решение или проверяемое состояние проекта не изменились.

Это правило действует во время рабочих сессий с доступом к файлам проекта. Оно не означает фоновое обновление документов без запущенной сессии.
